System Architecture

The prototype operates in a simple, serverless architecture using Google Workspace tools.

  1. Data Simulation (`simulator.gs`): A Google Apps Script function runs every minute to generate realistic machine data, including normal operations and fault scenarios.
  2. Data Storage (Google Sheets): The simulator writes this data into a structured Google Sheet (`AIM_MonitoringSystem.xlsx`) with multiple tabs for raw data, sensor readings, logs, and configuration.
  3. Business Logic (`anomaly.gs`, `alerts.gs`): Trigger-based Apps Script functions run automatically to analyze the new data for anomalies, send email alerts, and perform daily reporting and data maintenance.
  4. Data Reception (`receiver.gs`): A `doPost` web app function is available to receive data from external sources (like a Python script on an industrial PC in a real-world scenario).
  5. Visualization (Google Looker Studio): A Looker Studio dashboard is connected to the Google Sheet, providing a live, interactive user interface to visualize the machine status and historical data.

File Reference

simulator.gs

The main data source for the prototype. Generates simulated data for all machines and sensors on a timed trigger.

receiver.gs

Contains the `doPost(e)` web app function. This is the entry point for receiving external data via HTTP POST requests.

anomaly.gs

The core AI/ML engine. Contains logic to detect various types of anomalies, from simple threshold breaches to statistical deviations.

alerts.gs

Handles all user notifications. Sends real-time email alerts for new anomalies and a daily summary report.

maintenance.gs

Performs data housekeeping. Includes a function to automatically purge old raw data to keep the spreadsheet performant.

AIM_MonitoringSystem.xlsx

The Google Sheet that acts as the database for the entire system, containing all data and configuration.

Feature Details

Data Simulation

The `simulator.gs` file mimics a real-world production environment by generating data with built-in randomness. At each interval, a machine can be in one of four states: normal, fault, idle, or downtime, with configurable probabilities. Fault scenarios are pre-defined with specific sensor values and alarm codes to test the system's response.

Anomaly Detection

The `anomaly.gs` script runs a multi-rule check every minute:

  • Sensor Status: Logs anomalies for any sensor in a 'warning' or 'critical' state.
  • Value Drift: Compares the latest sensor value to its 30-reading average to detect slow drifts.
  • Cycle Time: Checks if machine cycle time exceeds warning or critical thresholds defined in the `Config` sheet.
  • Repeated Alarms: Flags alarms that repeat multiple times within a short window.
  • No Signal: Detects if a machine has stopped sending data while it was expected to be running.
  • Statistical Anomaly: Uses a 3-sigma rule to identify values that are statistically significant outliers from their recent history.

Alerting & Reporting

The `alerts.gs` file provides two key communication features:

  • Real-time Alerts: When `runAnomalyCheck()` finds new issues, it calls `sendAlerts()`. This function composes and sends an immediate HTML email detailing the specific anomalies, with color-coded risk scores.
  • Daily Reports: A separate function, `sendDailyReport()`, runs once a day to email a summary of the previous day's performance for all machines, including total downtime, alarm counts, and average cycle times.

Function Reference

receiver.gs

doPost(e): Webhook endpoint. Parses JSON data from an HTTP POST and writes to RawData, SensorData, and AlarmLog sheets. Updates MachineSummary.
getConfigValue(param): Utility function. Reads a specific parameter's value from the 'Config' sheet.

simulator.gs

runSimulator(): Main simulation loop. For each machine, decides a state (normal, fault, etc.), generates corresponding data, writes it to the database sheets, and calls `runAnomalyCheck()`.
seedHistoricalData(): One-time setup function. Populates the database with 7 days of historical data using the simulator logic.
startSimulator(): Creates and activates the 1-minute trigger for `runSimulator()`.
stopSimulator(): Finds and deletes the trigger for `runSimulator()`.

anomaly.gs

runAnomalyCheck(): The primary anomaly detection engine. Reads recent data and applies all defined rules to find and log anomalies.
setupAnomalyTrigger(): Creates the 1-minute trigger for `runAnomalyCheck()`.
isDuplicateAnomaly(...): A helper to prevent creating duplicate anomaly log entries in a short time.
runStatisticalChecks(...): A specific check for outliers based on a 3-sigma statistical rule.

alerts.gs

sendAlerts(): Reads `AnomalyLog` for un-sent items, formats an HTML email, and sends it via GmailApp.
sendDailyReport(): Gathers statistics for the previous day, builds a summary HTML email, and sends it.
setupDailyReportTrigger(): Creates the daily time-based trigger for the daily report.

maintenance.gs

cleanOldData(): Reads `DataRetention_days` from config and deletes rows in `RawData` and `SensorData` older than that.
setupCleanupTrigger(): Creates a weekly trigger to run the `cleanOldData` function.
